Theory of Coding, Episode 2: Maximum-Likelihood Decoding https://www.themathcitadel.com/theory-of-coding-episode-2-maximum-likelihood-decoding/ Sun, 27 Aug 2017 20:25:46 +0000 admin http://www.themathcitadel.com/?p=1000 The introduction to coding theory in this post will now allow us to explore some more interesting topics in coding theory, courtesy of Pinter's A Book of Abstract Algebra. We'll introduce the notion of a code, informations, and parity check equations. Most communication channels are noisy to some extent, which means that a transmitted codeword may have one or more errors. If the recipient of a codeword attempts to decode a message, he needs to be able to determine first if there was a possible error in the code, and then correct it. This will lead us to one method called maximum likelihood decoding.   Recall that we denoted [latex]\mathbb{B}^{n}[/latex] as the set of all binary words of length [latex]n[/latex]. We will call a subset of [latex]\mathbb{B}^{n}[/latex] a code. The first code we're going to work with is a subset of [latex]\mathbb{B}^{5}[/latex]. We'll call this code [latex]C_{1}[/latex], consisting of the following binary words with length 5: [latex, display = true]C_{1} = \left\{\begin{array}{lr}00000 & 10011\\00111 & 10100\\01001 & 11010\\01110 & 11101\end{array}\right\}[/latex] This is only a subset of [latex]\mathbb{B}^{5}[/latex]. Remember that the full cardinality of [latex]\mathbb{B}^{5} = 2^{5} = 32[/latex]. The 8 words in [latex]C_{1}[/latex] are codewords. Anything not in [latex]C_{1}[/latex] is not a codeword. When we design a code, only these codewords are to be transmitted. That means there is an error if any word not in [latex]C_{1}[/latex] was received. If we designed the code well, then it should be very unlikely that an error in transmission results in another codeword. (Think if we transmitted 10011 but received 11010. Since 11010 is a codeword, we wouldn't actually know that's not what was meant, and the transmission error would not be detected.) This means we need a good code that makes it easy on us to detect and correct transmission errors. To do that, we're going to need some notions of weight, distance, and redundancy.

Weight and Distance of Codewords

 
Definition: weight of a codeword: The weight of a binary word is the number of 1s in the word. Alternatively, we could add the bits (using regular addition.)
For example, the weight of 00111 is 3, and the weight of 01001 is 2. The position of the 1s is irrelevant; we're just counting them here.  
Definition: distance between codewords:  The distance between two binary words is the number of positions in which they differ
The distance between 00111 and 01001 is 3. (The two words differ in positions 2,3, and 4.)[note]The astute observer will notice that we can identify the positions in which the two words differ by using the XOR operation from the previous discussion.[/note]. We may also define the minimum distance of a code to be the smallest distance between any two words in the code. Looking up at the code [latex]C_{1}[/latex], we can make pairwise comparisons to see that the minimum distance of this code is 2.[note]There are 28 such comparisons to make. My suggestion: use the XOR operation from our previous discussion and then add bitwise to get the distance. A simple script can be written in your language of choice for good coding practice.[/note] What does this mean? When we receive a transmitted code over a potentially noisy channel, we will be able to notice we received a code with errors only if the number of errors in the transmitted code is less than this minimum distance; otherwise it is possible to receive a code word, not knowing that's not the code word that was meant. For instance, in [latex]C_{1}[/latex], we will be able to detect that we received a bad transmission if the number of bits that are in error is 1 or 0. It takes at least 2 errors in transmission before we could possibly be receiving another code word than what was sent. This gives us a way to design good codes. The more errors needed to change a codeword into another, the better the code is. Even more desirable would be a code with a minimum distance of 3, since 1 or 2 errors will always be detected, and 3 errors are highly unlikely.

Constructing a Code

In every codeword, certain positions are called information positions, and the remaining are redundancy positions, sometimes called parity bits. In [latex]C_{1}[/latex], the first three bits are information positions. Looking at the first three bits each of the eight code words, we can see that every possible 3-bit binary sequence is there: [latex, display = true]\{000,001,010,011,100,101,110,111\}[/latex] So what's up with the last two bits? How do we decide what those are?  For that, we use parity check equations. The simplest type of parity is single bit parity, and is either odd or even. That is, only the last position is a redundancy position. For odd parity, if the number of 1s in the binary word is even, then the redundancy position is set to a 1, giving a binary word (with parity) that has an odd number of ones. Otherwise, the parity bit is a 0. We reverse that for even single-bit parity. So if the word is 1100, under odd single-bit parity, the word with redundancy would be 11001. Under even single-bit parity, the word with redundancy is 11000. We can also define parity check equations when we want more than one bit of redundancy. For example, in [latex]C_{1}[/latex], the parity check equations are given by [latex, display = true]a_{4} = a_{1} + a_{3} \bmod 2 \qquad a_{5} = a_{1} + a_{2} + a_{3} \bmod 2[/latex] Check [latex]C_{1}[/latex] for yourself. The redundancy positions [latex]a_{4}[/latex] and [latex]a_{5}[/latex] for 110 are given by [latex, display = true] a_{4} = 1 \oplus 0 = 1 \text{ and } a_{5} = 1\oplus 1 \oplus 0 = 0[/latex] giving the word 11010 from the code above.[note] Feel free to test the other words as well[/note] We could create an entirely new code in [latex]\mathbb{B}^{5}[/latex] using the same information positions [latex]\mathbb{B}^{3}[/latex] and different parity check equations.

Decoding Noisy Communication: Maximum Likelihood Decoding

Let's make some shorthand for the distance between two binary words. We'll denote [latex]d(\mathbf{a},\mathbf{b})[/latex] to be the distance between two binary words. If we receive some transmission, we know it has to be made of words from our given code (in this case, we're working with [latex]C_{1}[/latex]). We decode a received word by finding the codeword closest to our received word. That is, if we receive [latex]\mathbf{x}[/latex], we minimize [latex]d(\mathbf{a},\mathbf{x})[/latex] over all the possible codewords [latex]\mathbf{a}[/latex] in our code. This is maximum likelihood decoding. If we receive 11111, then we need to find the codeword in [latex]C_{1}[/latex] closest. That is, the one that differs in the fewest positions. Likely you have already figured out that this word isn't a codeword, and thus is an erroneous transmission, and that the closest word is 11101. You did this quickly by calculating the minimum distance between 11111 and all the other words. Since it is not a codeword, and the minimum distance between codewords is 2, we know there must be at least one codeword that differs from 11111 by one position. Taking another example, let's assume we received 10111. This time, when you go through all the words, we actually find two codewords with a distance of 1 away: 10011 and 00111. That is, either a mistake could have been made in the 3rd position or the first position, but we actually can't tell which it is if we just receive the word (assuming no other context). This actually makes [latex]C_{1}[/latex] a less desirable code. The codewords are close enough together that we cannot ensure any possible word from [latex]\mathbb{B}^{5}[/latex] received can be uniquely decoded to a single word in [latex]C_{1}[/latex]

How do we find conditions to put on a code design to guarantee unique maximum-likelihood decoding?

[Fair warning here, we'll get a little more theoretical and math heavy. But here's where things get interesting.] Now it's time to get general. Let [latex]m[/latex] be the minimum distance for some general code [latex]C[/latex].[note]Doesn't matter the length of the binary words, just any code made of binary words[/note] First, let's formally state what we've already implicitly played with: as long as there are [latex]m-1[/latex] errors or fewer, we can detect an error. For instance, in [latex]C_{1}[/latex], [latex]m=2[/latex], and thus we can detect an error in one position, but not necessarily 2, since an error in two positions may give us another codeword that wasn't intended. But since we would decode it as a codeword, we wouldn't know that word wasn't the one our partner meant to send. The goal of this section is to figure out how to separate the codewords in any code to guarantee that any word received will be "closest" to only one word. In other words, the problem with the last code [latex]C_{1}[/latex] is that we found a word in [latex]\mathbb{B}^{5}[/latex] that was equidistant from two codewords. We want to design a code that keeps that from happening.    a and b are codewords in our general code [latex]C[/latex]. [latex]x, y, z[/latex] and [latex]w[/latex] are any other codewords of the same length as the codewords in [latex]C[/latex]. [latex]k[/latex] is the radius of the sphere drawn around each codeword. Here you can see that [latex]x[/latex] is in the sphere for a, and [latex]y[/latex] is in the sphere for b. [latex]z[/latex] and [latex]w[/latex] are outside of both. The spheres overlap, so in theory it's possible a codeword lies in both spheres. As shown in the figure above, we're going to define a sphere of radius k around a codeword [latex]\mathbb{a}[/latex] as the set of all words in [latex]\mathbb{B}^{n}[/latex][note]where n is the length of the codewords[/note] whose distance from [latex]\mathbb{a}[/latex] is [latex]k[/latex] or smaller. We'll call this set [latex]S_{k}(\mathbb{a})[/latex]. Mathematically, we write this as [latex, display = true]S_{k}(\mathbb{a}) := \{\mathbb{x} \in \mathbb{B}^{n} : d(\mathbb{a},\mathbb{x}) \leq k \}[/latex] [Quick aside on notation. The symbol [latex]:=[/latex] indicates a definition. The curly braces denote a set. The colon stands for the phrase "such that". So the above reads: The sphere of radius k around codeword a is defined as the set of all words of binary length n such that the distance from the word to a is less than or equal to the radius k.  You see here why mathematicians like our symbolic shorthand] If we want to design a good code, then we want to make sure that the spheres for all codewords are completely disjoint (not overlapping). So how big should we make these spheres in terms of the minimum distance of the code? If we let [latex]t = \frac{1}{2}(m-1)[/latex], then we can show that for any two spheres around generic codewords [latex]\mathbb{a}[/latex] and [latex]\mathbb{b}[/latex] in a generic code [latex]C[/latex] that have radius [latex]t[/latex], then the spheres will have no words in common. To show this, we'll show it by contradiction.[note]For all the mathematicians out there, bear with me. My goal is to impart intuition and a geometric reasoning, so I'm going to make a less formal proof here. The argument would center around looking at each position, via the definition of the minimum distance, but I want to relax formality here for clarity. The argument is no less correct, just less scary for non mathematicians. [/note] We'll assume that these two generic spheres have a word in common, say [latex]\mathbb{x}[/latex], and show that this assumption contradicts the notion of the minimum distance [latex]m[/latex]. If [latex]\mathbb{x}[/latex] is in both [latex]S_{t}(\mathbb{a})[/latex] and [latex]S_{t}(\mathbb{b})[/latex], then it must be at most [latex]t= \frac{1}{2}(m-1)[/latex] away from both [latex]\mathbb{a}[/latex] and [latex]\mathbb{b}[/latex]. Suppose it's directly in between; that is, suppose [latex]\mathbb{x}[/latex] is [latex]t[/latex] away from both [latex]\mathbb{a}[/latex] and [latex]\mathbb{b}[/latex], but still in the intersection. We know that the minimum distance is [latex]m[/latex], so the middle point between [latex]\mathbb{a}[/latex] and [latex]\mathbb{b}[/latex] is [latex]\frac{m}{2}[/latex]. But [latex]t = \frac{1}{2}(m-1)[/latex]. That means if [latex]\mathbb{x}[/latex] lies in [latex]S_{t}(\mathbb{a})[/latex], it can't be [latex]t[/latex] or less from [latex]\mathbb{b}[/latex], and vice versa. In other words, it's impossible for [latex]\mathbb{x}[/latex] to lie in the spheres of both codewords without contradicting the minimum distance [latex]m[/latex] between the two codewords. So what does this mean? This means that if we have [latex]t[/latex] or fewer errors in transmission, we will be able to uniquely decode [latex]\mathbb{x}[/latex], and won't run into the same problem we did with [latex]C_{1}[/latex].

Conclusion

Designing a code really relies on the parity check equations. The same information positions with different parity check equations yield totally different codes. We learned that some codes are better than others. Namely, we want a lot of "separation" between codewords, so we don't run into the issue of accidentally decoding the word incorrectly, but being unable to tell we decoded incorrectly. We want the minimum distance of the code to be as high as reasonable (3+ is considered reasonable, unless your transmission line is really bad). We also used a little geometry to view the notion of separation between codewords as spheres, and we want those spheres to not overlap each other. Overlapping spheres means that we can possibly decode a word incorrectly. Designing codes using parity check equations isn't an abstract exercise. We use it in data transmission all the time, but it requires a journey into the abstract a little bit to understand what makes some codes better than others. Score another point for algebra.
